Intro & Normativa

Fondamenti legali e obiettivi dell'accessibilità

Quadro Normativo

L'Italia adotta un sistema normativo gerarchico per l'accessibilità digitale, allineato alle direttive europee.

Legge Stanca (4/2004)

Prima normativa italiana sull'accessibilità. Ambito: siti web, app e documenti della PA.

Direttiva UE 2016/2102

Estende gli obblighi a tutti i siti e app mobile della PA in Europa.

Linee Guida AgID

Standard tecnico italiano basato su WCAG 2.1/2.2 (livello AA).

Dal 2025: L'European Accessibility Act (EAA) rende obbligatoria l'accessibilità anche per prodotti e servizi privati (es. e-commerce, bancomat).

Standard Obbligatori

WCAG 2.2 Livello AA

Standard obbligatorio per la PA italiana. Novità vs 2.1: maggiore attenzione a disabilità cognitive, mobile e drag-and-drop.

Principi POUR

  • Percepibile
  • Utilizzabile (Operable)
  • Comprensibile (Understandable)
  • Robusto

Inclusione Universale

Secondo ISTAT (2023), in Italia ci sono 3,1 milioni di persone con disabilità.

L'accessibilità non è solo un obbligo legale, ma un imperativo etico per garantire pari accesso all'informazione.

Sintesi degli Obblighi

1. Valutazione Tecnica

  • Verifica WCAG 2.2 AA
  • Autovalutazione o audit esterno
  • Frequenza: ogni 2 anni

2. Dichiarazione Accessibilità

  • Documento pubblico obbligatorio
  • Stato di conformità
  • Scadenza: 23 settembre/anno

3. Meccanismo di Feedback

  • Canale per segnalazioni
  • Risposta entro 20 giorni
  • Escalation: Difensore Civico
Responsabilità Editor: Applicare questi principi durante la pubblicazione degli articoli è responsabilità diretta di chi gestisce i contenuti.

Markup & Semantica

Dare significato alla struttura del documento

Gerarchia dei Titoli

La struttura semantica è fondamentale per chi utilizza Screen Reader. I titoli devono seguire un ordine logico e nidificato.

<!-- ✅ Corretto -->
<h1>Titolo Articolo</h1>
  <h2>Metodologia</h2>
    <h3>Campionamento</h3>
  <h2>Risultati</h2>
<!-- ❌ Errato: salto livello -->
<h1>Titolo</h1>
  <h4>Sezione</h4>
<!-- Salta h2 e h3! -->
Regola d'oro: Non saltare mai i livelli dei titoli solo per motivi estetici. Usa i CSS per lo stile!

Enfasi Semantica

✅ Semantico
È <strong>obbligatorio</strong> compilare.
Il termine <em>accessibilità</em>...
❌ Solo visivo
È <b>obbligatorio</b> compilare.
Il termine <i>accessibilità</i>...

Perché preferire strong/em?

  • Gli screen reader cambiano tono ed enfasi vocale
  • <b> e <i> sono solo stilistici
  • Migliore SEO: i motori pesano il contenuto in <strong>

Liste Semantiche

Usare i tag appropriati per gli elenchi, mai simularli con trattini manuali.

<!-- Lista non ordinata -->
<ul>
  <li>Elemento</li>
</ul>
<!-- Lista ordinata -->
<ol>
  <li>Primo</li>
</ol>
<!-- Definizioni -->
<dl>
  <dt>Termine</dt>
  <dd>Def.</dd>
</dl>
Perché? Gli screen reader annunciano "lista di 5 elementi" e permettono di navigare tra gli item. Con i trattini manuali, l'utente sente solo "trattino primo elemento trattino..."

Citazioni

<!-- Citazione in blocco -->
<blockquote cite="url">
  <p>Il potere del Web...</p>
  <footer>— Tim B-Lee</footer>
</blockquote>
<!-- Citazione inline -->
Come disse <q>...</q>
<!-- Titolo opera -->
Nel documento <cite>WCAG 2.2</cite>

Differenze:

  • <blockquote> → citazioni lunghe, blocco separato
  • <q> → citazioni brevi inline (aggiunge virgolette auto)
  • <cite> → titolo di un'opera
❌ Errato: virgolette e corsivo manuale
<i>"Il potere..."</i>

CSS & Layout

Gestione estetica accessibile del tema

Specificità CSS

Quando più regole si applicano allo stesso elemento, vince quella con specificità maggiore.

SelettorePesoNote
!importantMassimoUltima risorsa, evitare
Stili inline1000Evitare se possibile
ID #id100Un solo elemento/pagina
Classi .class10✅ Metodo consigliato
Tag p, div1Stili di base
Consiglio: Prima di usare !important, prova ad aumentare specificità combinando selettori: .content .intro { }

Selettori e Contesto

Come applicare uno stile solo in un contesto specifico?

/* ❌ Troppo generico */
.title {
  color: red;
}

/* ✅ Contestuale */
article.details .title {
  color: red;
}
.summary .title {
  color: blue;
}

Vantaggi:

  • Specificità maggiore senza !important
  • Stili isolati per contesto
  • Nessun conflitto
  • Codice più manutenibile

Colori e Contrasti

Il contrasto deve essere sufficiente per ipovedenti e in condizioni di forte luce.

TipoRapporto minimo
Testo normale4.5:1
Testo grande (≥18pt)3:1
Elementi UI/grafici3:1

Tool di verifica:

  • WebAIM Contrast Checker
  • Colour Contrast Analyser
  • Firefox DevTools (tab Accessibilità)
Attenzione: Evitare sfondi immagine sotto il testo. Il testo su immagini è quasi sempre problematico per il contrasto.

Tipografia e Spazi

Usa unità relative (rem)

/* ❌ Pixel: fisso */
body { font-size: 16px; }

/* ✅ Rem: si adatta */
body { font-size: 1rem; }
h1 { font-size: 2rem; }

Conversione: 12px=0.75rem · 16px=1rem · 24px=1.5rem

Best practice

  • line-height: 1.5-1.6 per facilitare la lettura
  • text-align: left, mai justify (spazi irregolari problematici per dislessici)
p {
  line-height: 1.6;
  text-align: left;
}

Contenuti & Asset

Immagini, Link e File

Gestione Immagini

Ogni immagine deve avere un attributo alt descrittivo. Se decorativa, alt="" (vuoto, non assente).

✅ Corretto
<img src="grafico.png"
  alt="Vendite 2024: Q1 120k, Q2 145k...">

<img src="decoro.png" alt="">
❌ Errato
<img src="IMG_20240601.jpg">
<!-- Alt assente! -->

<img src="foto.jpg" alt="foto">
<!-- Alt inutile -->
Regole: Descrivi contenuto e scopo, max ~125 caratteri, non iniziare con "Immagine di...". Comprimere le immagini per il web (72-96 DPI, <200KB).

Gestione Link

Il testo del link deve essere auto-esplicativo: capire la destinazione senza leggere il contesto.

❌ Evitare✅ Preferire
"clicca qui""Scarica il PDF della ricerca"
"leggi tutto""Consulta la bibliografia completa"
"maggiori info""Informazioni sul bando 2025"
target="_blank": Aprire in nuova scheda senza avvisare disorienta. Preferire stessa scheda o avvisare esplicitamente.
<a href="doc.pdf">
  Report 2024 (PDF, 2.3MB)
</a>

Gestione File

PDF ottimizzati

  • Usa "Salva per Web" o "Ottimizza PDF"
  • 72-150 DPI (non 300 DPI da stampa)
  • Peso: <5MB standard, <10MB complessi

Indica sempre formato e peso

"Scarica il bando (PDF, 1.2MB)"

Accessibilità PDF

  • I PDF devono essere "tagged" (strutturati) per screen reader
  • Evitare PDF da scansione (immagini di testo)
  • Se necessario: applicare OCR

User Experience

Semplicità e chiarezza dei contenuti

UX e Semplicità

L'accessibilità non riguarda solo il codice, ma anche come scriviamo. Una buona UX riduce il carico cognitivo.

❌ UX Complessa

Muri di testo compatti, paragrafi infiniti, linguaggio inutilmente burocratico, nessuna gerarchia visiva.

✅ UX Semplice

Frasi brevi, paragrafi ben distanziati, liste puntate, grassetti per i concetti chiave.

Consiglio dell'esperto: Applica la tecnica della "Piramide Rovesciata": inserisci le conclusioni o le informazioni più importanti all'inizio di ogni capitolo.

Tool di Controllo

Verificare il lavoro svolto

Strumenti di Audit

Lighthouse (Chrome)

Strumento gratuito integrato in Google Chrome.

  1. Aprire pagina in Chrome
  2. F12 → Tab "Lighthouse"
  3. Selezionare "Accessibility"
  4. Cliccare "Analyze page load"

Report con punteggio 0-100 e suggerimenti.

Mauve++ (CNR)

Validatore specifico per linee guida italiane AgID.

  1. Accedere a mauve.isti.cnr.it
  2. Inserire URL da verificare
  3. Avviare scansione
  4. Consultare report per criterio WCAG

Esporta in PDF/CSV per documentazione.
